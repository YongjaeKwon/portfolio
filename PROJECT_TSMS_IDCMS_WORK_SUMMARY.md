# TSMS / IDCMS / AS 업무 화면 정리

> 공개 포트폴리오 반영 전 정리 문서입니다. 내부 코드, 운영 URL, 기관별 상세 정책, 보안 설정값은 제외했습니다.

## 한 줄 요약

AS 접수, 상담, 후송/배정, 모바일 접수, 무인보관함 접수, 개인정보 동의, QR 확인, 알림톡 발송처럼 현장 업무와 외부 연동이 섞이는 화면을 WebSquare/JSP/Java 기반으로 개발하고 운영 이슈를 개선했습니다.

## 내가 맡은 역할

- AS 접수 및 상담 화면의 화면/쿼리/로직 개선
- 모바일 및 태블릿 환경에서 쓰이는 접수, 확인, 서명, QR 화면 처리
- 개인정보 동의 및 QR 확인 흐름에서 공개 페이지, 토큰, 조회 조건, 저장 조건 처리
- 알림톡 발송, 서명 확인, 발송 후 상태 잠금 등 외부 발송 결과와 화면 상태 연결
- 교육청/학교/센터/CE 조건에 따라 조회 범위와 처리 가능 액션이 달라지는 화면 로직 개선

## 주요 업무 내용

### 1. AS 후송 및 배정 처리

AS 세부 진행 화면에서 제조사 후송, 센터 간 후송, 본사 후송 등 후속 처리 조건을 개선했습니다.

- 로그인한 부서와 CE 기준 자동 배정 조건 처리
- 서비스의 부서 코드와 로그인 CE 코드가 맞지 않을 때 제조사 후송을 제한하는 조건 처리
- 부서 코드 선택 시 CE 선택 팝업이 이어지는 흐름 개선
- AS 배정 시 센터 검색이 가능하도록 화면 상태 변경

포트폴리오 표현 후보:

> AS 후속 처리 화면에서 제조사/센터/본사 후송 조건을 정리하고, 로그인 사용자와 서비스 부서/CE 조건에 따라 가능한 액션이 달라지도록 화면과 데이터 조회 흐름을 개선했습니다.

### 2. AS 접수 및 상담 화면

AS 접수와 상담 화면에서 입력 화면, 조회 조건, 처리 로직을 개선했습니다.

- AS 접수 화면 추가
- 상담 및 AS 접수 화면 변경
- 화면 업그레이드와 함께 쿼리 및 로직 수정
- 우리학교 패드 현황 화면 기능 부착
- 학년/반/번호 비교 오류가 있던 조회 조건 수정

포트폴리오 표현 후보:

> AS 접수와 상담 화면에서 입력/조회 화면을 추가하고, 학교/학급/기기 기준 조회 조건이 실제 업무 기준과 맞도록 화면과 SQL 조건을 함께 수정했습니다.

### 3. 모바일 무인보관함 접수

모바일에서 진행되는 무인보관함 접수 화면과 백엔드 처리 흐름을 개발했습니다.

- 무인보관함 접수 화면 세트 생성
- QR 스캔, 번호 발송, 접수 단계별 화면 구성
- 접수 완료 화면과 안내 이미지/버튼 구성
- 접수 정보 저장을 위한 Controller, Service, DAO, Mapper 추가
- QR 접수 시 센터 배정 조건 추가
- 무인보관함 현황 조회에서 null 조건 처리
- 무인보관함 관리 화면 엑셀 다운로드 추가

포트폴리오 표현 후보:

> 모바일 무인보관함 접수 화면에서 QR 스캔, 번호 발송, 접수 진행, 완료 화면까지 이어지는 단계를 구성하고, 접수 저장 및 센터 배정 조건을 백엔드와 함께 구현했습니다.

### 4. 개인정보 동의 및 QR 확인

교육청 노트북/기기 대여 동의 화면에서 QR 기반 확인, 기존 정보 조회, 설치 일정 팝업, 공개 페이지 접근 흐름을 처리했습니다.

- 개인정보 동의 화면 신규/기존 연도 분리
- 공개 접근용 Controller와 URL-safe 토큰 유틸 추가
- 학생/기기 토큰 기반 QR 확인 흐름 구성
- QR 확인 화면에서 기존 정보와 신규 입력 정보 비교
- QR 다크모드 대응용 배경 처리
- 기존 정보 팝업, 유의사항, 헤더 연도 표시 개선
- 설치 일정 팝업 추가
- QR 확인 시 배부 정보 조회 조건 추가
- 삭제 이력 테이블 및 조회 컬럼 보완

포트폴리오 표현 후보:

> 개인정보 동의 화면에서 QR 기반 확인, 기존 정보 조회, 설치 일정 팝업, 공개 URL 접근 토큰 처리를 구현해 모바일 사용자가 동의와 배부 정보를 확인할 수 있는 흐름을 구성했습니다.

### 5. 배송희망일 및 일정 등록

배송희망일 등록 화면, 완료 화면, 마감 처리, 일정 등록 화면의 예외 처리를 개선했습니다.

- 배송희망일 등록 화면과 완료 화면 생성
- 날짜 목록 및 마감 처리 조건 변경
- 발송일 기준 마감 조건 추가
- 뒤로가기 방지 처리
- 데이터가 없는데 파싱되는 경우 오류 화면으로 전환
- 일정관리 페이지에서 월 변경 시 날짜 선택 오류 수정
- 모바일 화면에서도 일정 등록이 가능하도록 화면 보완

포트폴리오 표현 후보:

> 배송희망일 등록 화면에서 날짜 선택, 마감 조건, 완료 화면, 예외 화면을 정리해 사용자가 잘못된 일정으로 접수하지 않도록 처리했습니다.

### 6. 알림톡 및 발송 연동

알림톡 발송 관리 화면에서 발송 API, 발송 후 화면 잠금, 서명 확인 팝업, 템플릿 확인 기능을 처리했습니다.

- 알림톡 발송 Controller/Service/DAO/Mapper 연동
- 발송 완료 시 선택 항목 잠금 처리
- 알림톡 템플릿 서명 확인 팝업 생성
- 서명 확인 시 사용 여부 표기 및 출력 기능 추가
- 배송 일정 등록 화면과 알림톡 발송 조건 연결

포트폴리오 표현 후보:

> 알림톡 발송 화면에서 발송 요청, 발송 후 선택 잠금, 템플릿 서명 확인 팝업을 추가해 외부 발송 결과가 화면 상태와 어긋나지 않도록 개선했습니다.

### 7. 태블릿 전자서명

태블릿 환경에서 쓰이는 서명 접수 화면을 보완했습니다.

- 태블릿 화면 정의 및 리소스 갱신
- 서명 접수 완료 화면 처리
- 팝업 메시지와 사인 화면 변경
- 모바일/태블릿 업무 흐름에서 같은 접수 정보가 이어지도록 화면 정의 수정

포트폴리오 표현 후보:

> 태블릿 전자서명 화면에서 서명 완료, 팝업 메시지, 화면 정의를 보완해 모바일 접수 이후의 후속 단계가 끊기지 않도록 처리했습니다.

## 기술적 난이도

- WebSquare 화면, JSP 공개 페이지, Java Controller/Service/DAO, MyBatis Mapper가 함께 움직이는 구조였습니다.
- 모바일/태블릿/관리자 화면이 같은 업무 번호와 같은 사용자 정보를 기준으로 이어져야 했습니다.
- 개인정보 동의, QR 확인, 알림톡 발송처럼 보안과 외부 연동이 포함된 기능은 실패/예외 상태를 화면에 분리해서 보여야 했습니다.
- 운영 중인 화면을 수정하면서 기존 기관별 정책과 신규 연도 정책을 함께 유지해야 했습니다.

## 아키텍처 요약

```text
모바일/태블릿/관리자 화면
  -> WebSquare XML/JS 또는 JSP 공개 페이지
  -> Java Controller
  -> Service / DAO
  -> MyBatis Mapper
  -> 업무 DB
  -> 알림톡/QR/토큰/파일/엑셀 등 외부 또는 유틸 처리
```

주요 구성:

- Frontend: WebSquare, JSP, JavaScript, 태블릿/모바일 화면
- Backend: Java, Spring MVC 계열 Controller/Service/DAO
- Database: MyBatis Mapper, 기관/사용자/기기 기준 조회 조건
- External/Utility: QR, URL-safe token, BizTalk/알림톡, Excel, mobile public page
