# 프로젝트별 면접 답변 초안

> 목적: 포트폴리오를 보고 들어온 질문에 "제가 한 일은..."으로 바로 답할 수 있도록 준비한 초안입니다. 실제 면접에서는 회사명이나 내부 정책보다 업무 흐름, 담당 범위, 검증 방식을 중심으로 말하는 것이 안전합니다.

## PPS 협력사 포탈

### 1. 가장 자신 있게 설명할 수 있는 기능은 무엇인가요?

PPS에서는 교육관리 기능을 가장 자신 있게 설명할 수 있습니다. 교육을 등록한 뒤 대상자를 엑셀로 업로드하고, 대상자별 안내 메일을 발송한 다음 제출 현황을 조회하는 흐름을 구현했습니다. 단순히 화면만 만든 것이 아니라 Vue 화면 상태, Spring Boot API 파라미터, MyBatis SQL 조회 조건이 같은 대상자 기준으로 이어지는지 확인하며 개발했습니다.

### 2. 공통화 경험이 있나요?

게시판, 제안하기, 설문 화면에서 댓글과 대댓글 기능이 반복되고 있었습니다. 화면마다 따로 구현하면 수정 시 누락이 생길 수 있어 댓글 조회, 대댓글 조회, 등록, 수정, 삭제 흐름을 공통 Vue 컴포넌트로 분리했습니다. 이후 여러 화면에서 같은 방식으로 댓글 기능을 사용할 수 있도록 정리했습니다.

### 3. 운영 이슈를 어떻게 확인했나요?

운영 이슈가 들어오면 먼저 화면에서 어떤 조건으로 문제가 재현되는지 확인하고, 그 조건이 API 파라미터와 MyBatis SQL에 동일하게 전달되는지 봤습니다. 교육 대상자, CE 검증 상태, 협력사 검색처럼 조건이 많은 화면은 필터 하나가 빠져도 결과가 달라질 수 있어서 화면, API, SQL을 함께 점검했습니다.

## TSMS / IDCMS AS 업무 시스템

### 1. 이 프로젝트에서 본인의 역할은 무엇이었나요?

AS 접수, 개인정보 동의, QR 확인, 태블릿 전자서명, 진행상태 조회처럼 사용자가 실제로 거치는 업무 화면을 주로 담당했습니다. WebSquare/JSP 기반 화면을 개발하고 Java/Spring MVC/MyBatis 조회 조건을 함께 확인했습니다. 모바일, 태블릿, 데스크톱 화면이 같은 업무 번호와 상태를 기준으로 이어지도록 구현한 점이 핵심입니다.

### 2. 어려웠던 점은 무엇인가요?

접수 이후 동의, 첨부, 서명, 알림 발송, 상태 조회가 여러 화면으로 이어지기 때문에 중간 단계의 상태가 어긋나면 사용자가 다음 업무로 넘어가지 못하는 문제가 생길 수 있었습니다. 그래서 필수값 누락, 동의 미완료, 첨부 실패, 서명 저장 실패, 알림톡 발송 실패 같은 상태를 나눠 화면에 반영했습니다.

### 3. 어떻게 검증했나요?

업무 번호를 기준으로 모바일 접수, 개인정보 동의, 전자서명, 진행상태 조회가 같은 데이터를 보고 있는지 확인했습니다. 또한 권한이나 센터 조건이 바뀌었을 때 목록과 상세 화면의 조회 범위가 일치하는지 점검했고, 알림톡/BizTalk 발송 결과가 성공, 실패, 대기 상태에 따라 화면에 다르게 표시되는지 확인했습니다.

## 교육청 IT 자산관리

### 1. 이 프로젝트의 핵심은 무엇이었나요?

교육청, 학교, 부서 권한에 따라 볼 수 있는 자산 범위가 달라지는 점이 핵심이었습니다. 자산 대시보드, 자산현황, 유상처리 현황 화면에서 기관, 사업 차수, 자산 상태 기준의 조회 조건을 처리했습니다. 화면 필터와 MyBatis SQL 조건이 같은 기준으로 동작하도록 점검했습니다.

### 2. 대시보드 기능에서 신경 쓴 부분은 무엇인가요?

대시보드는 집계 수치를 먼저 보여주지만, 사용자는 그 수치를 클릭해 상세 목록으로 들어갑니다. 그래서 대시보드 카드의 집계 기준과 클릭 이후 목록 조회 조건이 어긋나지 않도록 확인했습니다. 사업 차수, 기관, 자산 상태 조건이 목록과 통계에 동일하게 적용되는지를 중점적으로 봤습니다.

### 3. 권한 조건은 어떻게 처리했나요?

화면에서만 제한하는 방식은 위험하다고 생각했습니다. 교육청, 학교, 부서 권한별로 검색 조건이 SQL 조회 조건까지 반영되어야 실제 데이터 노출 범위를 제어할 수 있기 때문입니다. 권한별 계정 기준으로 목록, 상세, 대시보드 집계 결과가 같은 기준으로 제한되는지 확인했습니다.

## SR30 물류관리시스템

### 1. SR30은 어떤 경험으로 설명하는 것이 좋나요?

SR30은 최신 프론트엔드 기술보다는 운영 시스템의 조건과 통계를 정확히 맞추는 경험으로 설명하는 것이 좋습니다. 일정, 설문, 물류, 재고, 서비스 리포트, KPI, 엑셀 다운로드, 관리자 이력 조회 화면을 개선하면서 화면 필터, SQL 조건, 다운로드 결과가 같은 기준으로 동작하는지 확인했습니다.

### 2. 설문 기능에서 담당한 부분은 무엇인가요?

설문 등록, 대상자 관리, 응답 저장, 통계 조회 흐름을 담당했습니다. 운영자가 대상자를 조회하고 응답 상태를 확인한 뒤 통계 결과를 볼 수 있도록 화면과 SQL 조건을 정리했습니다. 응답 상태와 통계 집계 기준이 다르면 결과가 달라질 수 있어 화면 조건과 집계 SQL을 함께 확인했습니다.

### 3. 레거시 기술 경험은 어떻게 설명하면 좋나요?

Nexacro 같은 레거시 환경을 사용했지만, 핵심은 특정 도구보다 운영 화면의 기준을 맞추는 경험이었다고 설명하면 좋습니다. 화면 조회, 통계 집계, 엑셀 다운로드, 이력 조회가 같은 조건으로 동작해야 운영자가 데이터를 신뢰할 수 있습니다. 이 경험은 Vue나 React 기반 관리자 화면에서도 그대로 적용할 수 있는 업무 이해 역량입니다.

## 공통 질문 답변

### 프론트엔드 개발자인데 왜 SQL이나 백엔드 흐름을 봤나요?

운영 시스템에서는 화면에 보이는 결과가 대부분 조회 조건과 권한 조건에 의해 결정됩니다. 화면만 고치면 당장은 동작해 보여도 SQL 조건이 빠져 있으면 다른 권한이나 다른 기간에서 문제가 재현될 수 있습니다. 그래서 프론트엔드 구현을 할 때도 API 응답과 MyBatis SQL 흐름을 함께 확인하며 화면 요구사항과 실제 데이터가 맞는지 점검했습니다.

### 레거시 경험이 많은데 최신 프론트엔드도 가능한가요?

실무에서는 Vue, WebSquare, JSP, Nexacro 등 여러 환경을 다뤘고, 개인/팀 프로젝트에서는 React, TypeScript, Next.js, Zustand, React Query를 사용했습니다. 실무에서 쌓은 강점은 복잡한 업무 흐름과 상태 조건을 이해하는 능력이고, 최신 프론트엔드 프로젝트를 통해 타입, 상태 관리, API 연동 구조를 계속 보강하고 있습니다.

### 본인의 차별점은 무엇인가요?

단순히 화면을 만드는 것보다 사용자가 실제로 처리하는 업무 흐름을 이해하고 화면, API, SQL, 외부 연동 결과가 끊기지 않도록 확인하는 점입니다. 등록, 업로드, 발송, 조회, 이력 확인처럼 여러 단계가 이어지는 운영 화면을 개발해왔고, 권한이나 상태값 때문에 생기는 예외를 끝까지 추적하는 데 강점이 있습니다.
