# 포트폴리오 개선 방향 및 실행 로드맵

> 목적: 지금까지의 실무 경험을 "많이 했다"가 아니라 "운영 업무를 이해하고 화면, API, SQL, 외부 연동까지 연결해 문제를 해결한 프론트엔드 개발자"로 보이게 만드는 것.

## 최종 포지셔닝

권장 포지션 문장:

> B2B/B2G 운영 시스템에서 관리자 화면, API 연동, 권한별 조회 조건, MyBatis SQL, 메일/파일/인증 연동까지 함께 다뤄온 프론트엔드 개발자입니다. 단순 화면 구현보다 실제 사용자가 반복적으로 처리하는 업무 흐름을 이해하고, 등록, 업로드, 발송, 조회, 이력 확인이 끊기지 않도록 구현하는 데 강점이 있습니다.

이 방향이 좋은 이유:

- 현재 이력은 예쁜 UI보다 업무 복잡도와 운영 화면 경험이 강점입니다.
- PPS, TSMS/IDCMS, IT 자산관리, SR30 모두 "업무 흐름을 화면으로 구현"한 공통점이 있습니다.
- 프론트엔드 지원 시에도 API, SQL, 권한 조건을 함께 이해했다는 점은 차별점이 됩니다.
- 다만 레거시 기술만 한 사람처럼 보이지 않도록 Vue/React/TypeScript 쪽 보강이 필요합니다.

## 프로젝트 우선순위

### 1순위: PPS 협력사 포탈

대표 프로젝트로 가장 앞에 배치합니다.

강점:

- Git 커밋 근거가 가장 풍부합니다.
- Vue, Spring Boot, MyBatis, 메일, 엑셀, OTP, 공통 컴포넌트가 함께 있습니다.
- 교육관리, CE 현황, 게시판/설문 댓글 공통화처럼 설명하기 좋은 업무 단위가 많습니다.

포트폴리오에서 보여줄 핵심:

- 교육 등록 -> 대상자 엑셀 업로드 -> 메일 발송 -> 제출 현황 조회 흐름
- 댓글/대댓글 공통 컴포넌트화
- CE 현황 메일 발송 및 발송 이력 저장
- 공지 읽음 이력, 협력사/센터 검색, OTP 예외 처리

면접 스토리 후보:

- "단순 화면 추가가 아니라 운영자가 반복 처리하는 교육 업무 흐름을 하나의 관리 흐름으로 연결했다."
- "댓글 기능이 여러 화면에 흩어져 있어 공통 Vue 컴포넌트로 분리했다."
- "메일, 엑셀, OTP처럼 실패 케이스가 생기기 쉬운 연동 기능에서 화면 상태와 서버 처리 흐름을 같이 점검했다."

### 2순위: TSMS/IDCMS/AS 업무

두 번째 대표 프로젝트 또는 상세 case study로 사용합니다.

강점:

- 모바일 접수, QR 동의, 태블릿 전자서명, 무인보관함 등 사용자가 직접 마주하는 업무 흐름이 있습니다.
- 운영 시스템이지만 사용자 접점이 명확해 설명력이 좋습니다.
- 알림톡/BizTalk, 일정, 배송, 접수 같은 실제 현장 업무와 연결됩니다.

포트폴리오에서 보여줄 핵심:

- AS 접수/이관/배정 흐름
- 개인정보 동의 및 QR 기반 확인 화면
- 모바일/태블릿 전자서명 흐름
- 알림톡/BizTalk 발송 및 상태 처리

면접 스토리 후보:

- "현장 접수와 관리 화면이 같은 상태를 바라보도록 접수, 동의, 서명, 알림 흐름을 연결했다."
- "사용자 입력이 실제 처리 상태와 어긋나지 않도록 화면 조건과 서버 조회 조건을 함께 확인했다."

### 3순위: 교육청 IT 자산관리

권한, 상태, 대시보드 중심 프로젝트로 사용합니다.

강점:

- 교육청, 학교, 부서 권한에 따른 조회 범위 처리를 설명하기 좋습니다.
- 자산 상태와 대시보드, 유상처리 현황, 배치 이력 등 운영 관리 성격이 강합니다.
- B2G 시스템 경험을 보여주기 좋습니다.

포트폴리오에서 보여줄 핵심:

- 자산 등록/현황/상태 관리
- 교육청/학교/부서별 권한 조건
- 대시보드 클릭 라우팅 및 현황 조회
- 배치 이력과 유상처리 현황 화면

면접 스토리 후보:

- "권한별로 보이는 데이터가 달라지는 B2G 업무에서 화면 필터와 SQL 조건이 일치하도록 구현했다."
- "상태값이 많은 자산 관리 업무에서 목록, 상세, 대시보드의 기준이 어긋나지 않도록 점검했다."

### 4순위: SR30 물류관리시스템

보조 프로젝트로 두되, 넥사크로/레거시 경험만 강조되지 않게 정리합니다.

강점:

- 변경 이력이 많고 업무 범위가 넓습니다.
- 일정, 설문, 물류, 재고, KPI, 리포트, 개인정보 이력 등 다양한 운영 화면 경험을 보여줄 수 있습니다.

주의점:

- 너무 많이 쓰면 포트폴리오가 산만해집니다.
- Nexacro 중심으로 보이면 최신 프론트엔드 지원에서 매력이 약해질 수 있습니다.
- "운영 시스템 개선 경험"과 "복잡한 업무 조건 처리 경험"으로 요약하는 편이 좋습니다.

포트폴리오에서 보여줄 핵심:

- 일정 공유 및 근무 일정 관리
- 설문 등록/응답/통계 흐름
- 물류/재고/입출고/리포트 화면
- KPI 및 관리자 이력 조회

### 보강 프로젝트: ddoing / SSAFAST / MODAC / quant-core

실무 프로젝트 4개를 중심으로 두고, 최신 프론트엔드 역량을 보강하는 프로젝트로 사용합니다.

ddoing:

- React, TypeScript, Redux, Canvas, AI 추론 API 연동 경험을 보여주는 보강 프로젝트로 사용합니다.
- "AI 모델을 만들었다"보다 "AI 추론 결과를 Drawing Page의 게임 흐름과 사용자 상태로 연결했다"는 방향으로 설명합니다.
- PM 역할, 학습 데이터 전처리 참여, Main Page/Drawing Page 구현, Hall of Fame/My Page 개선을 핵심으로 둡니다.

SSAFAST:

- Next.js, TypeScript, React Hook Form, React Query 기반 개발자 도구형 UI 경험을 보여줍니다.
- 수상 이력이 있어 SSAFY 프로젝트 중 가장 강한 보조 근거로 유지합니다.

MODAC:

- Vue 3, Pinia, WebSocket, Markdown, GitHub commit 연동 경험을 보여주는 보조 프로젝트로 사용합니다.
- PPS와 Vue 계열이 겹치므로 실무 프로젝트보다 뒤에 배치하되, 별도 카드로 노출하고 GitHub 저장소로 바로 연결합니다.

quant-core:

- 현재 학습 중인 React/TypeScript/FastAPI/Docker 기반 개인 프로젝트로, 최신 기술 보강용으로 유지합니다.
- 완성도가 더 올라가기 전까지는 실무 프로젝트보다 앞에 두지 않습니다.

## 2차 실행 로드맵: 1주

목표: 단순 경력 나열을 넘어 "면접에서 설명 가능한 포트폴리오"로 만든다.

### 1. 대표 프로젝트 2개를 깊게 만든다

추천 조합:

- PPS: 현대 웹 포탈, Vue, Spring Boot, MyBatis, 메일/엑셀/OTP
- TSMS/IDCMS: 모바일/태블릿, QR, 서명, 알림톡, 현장 업무 흐름

해야 할 일:

- 각 프로젝트별 핵심 업무 흐름 1개를 그림 없이 텍스트 구조로 정리
- 화면 -> API -> Service -> SQL -> 외부 연동 순서로 설명 가능하게 준비
- "어려웠던 점"과 "어떻게 검증했는지"를 각 2개씩 준비

면접 답변 템플릿:

> 이 기능은 운영자가 반복적으로 처리하던 업무를 줄이기 위한 기능이었습니다. 저는 화면 입력과 조회 조건을 구현하고, Spring Controller/Service와 MyBatis SQL까지 함께 확인해 실제 저장/조회 결과가 화면 상태와 일치하도록 작업했습니다. 특히 권한, 상태값, 기간 조건에 따라 결과가 달라지는 부분을 중점적으로 검증했습니다.

### 2. 정량 표현을 안전하게 정리한다

사용해도 좋은 수치:

- PPS 개인 커밋 268건
- PPS merge 제외 커밋 216건
- PPS 주요 변경 파일 642건
- SR30 SVN 작업 리비전 109건
- IDCMS/TSMS 관련 SVN 작업 리비전 97건
- IT 자산관리 SVN 작업 리비전 20건

표현 방식:

- "성과"가 아니라 "참여 규모" 또는 "작업 근거"로 사용합니다.
- 예: "Git/SVN 이력 기준 200건 이상의 기능 개선 및 운영 이슈 대응에 참여"
- 예: "관리자 화면, API, SQL, 메일/파일 연동을 포함한 운영 기능을 지속적으로 개선"

주의할 수치:

- 응답률 5% -> 20% 개선
- 처리 시간 단축
- 오류율 감소
- 사용자 수, 트래픽, 매출, 비용 절감

위 수치는 실제 집계 자료가 없으면 쓰지 않습니다.

### 3. 기술 키워드를 재배치한다

현재 강점:

- Vue.js
- JavaScript
- Spring Boot / Spring MVC
- MyBatis
- JSP / WebSquare / Nexacro
- Oracle / MariaDB
- 업무 화면, 관리자 화면, 권한, 상태, 파일, 메일, 알림

보강해야 할 키워드:

- TypeScript
- React 또는 Vue 3 Composition API
- 상태 관리
- 테스트
- 접근성
- 성능 최적화
- API 계약 이해

전략:

- 실무 프로젝트는 업무 복잡도 중심으로 보여줍니다.
- 개인 프로젝트는 최신 프론트엔드 역량 보강용으로 사용합니다.

## 3차 실행 로드맵: 2~4주

목표: "실무는 레거시 위주인가?"라는 우려를 줄이고, 최신 프론트엔드 채용에도 대응한다.

### 보강 프로젝트 방향

새 프로젝트를 만든다면 디자인보다 기능 완성도를 우선합니다.

추천 주제:

> 운영 관리자용 실시간 업무 대시보드

기능:

- 로그인 또는 권한별 메뉴 분기
- 목록/상세/등록/수정 CRUD
- 검색 조건, 기간 필터, 상태 필터
- 엑셀 다운로드 또는 CSV 다운로드
- 메일 발송 모의 기능
- 처리 이력 타임라인
- API loading/error/empty 상태 처리
- TypeScript 타입 정의
- 테스트 코드 일부 작성

기술:

- React + TypeScript 또는 Vue 3 + TypeScript
- TanStack Query 또는 Pinia
- MSW 또는 간단한 mock API
- Vitest
- Playwright smoke test

이 프로젝트의 목적:

- 실무에서 해온 관리자 화면 경험을 최신 프론트엔드 기술로 다시 증명합니다.
- 포트폴리오의 레거시 비중을 낮춥니다.
- 면접에서 "실무는 JSP/Nexacro도 했지만, 최신 스택으로는 이렇게 구현할 수 있다"라고 말할 수 있습니다.

완료 기준:

- README에 설치/실행/테스트 방법이 있어야 합니다.
- 핵심 화면 3개 이상이 실제로 동작해야 합니다.
- 테스트가 최소 3개 이상 있어야 합니다.
- 포트폴리오에는 "개인 프로젝트"가 아니라 "실무 경험을 현대 프론트엔드 방식으로 재구성한 프로젝트"로 소개합니다.

### SI/SM, B2B 솔루션, 사내 시스템 회사
