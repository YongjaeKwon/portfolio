# 교육청 자산관리 시스템

> 공개용 정리입니다. 내부 코드, 고객사별 상세 정책, 운영 URL, 계정, 환경 정보는 제외했습니다.

- 문제: 권한별 조회 범위와 대시보드 집계 기준이 어긋나면 운영 수치가 달라질 수 있었습니다.
- 담당: JSP 화면, Spring MVC API, MyBatis SQL, 대시보드 집계와 현황 목록 조건을 함께 수정했습니다.
- 결과: 권한, 기관, 자산 상태 기준이 대시보드와 상세 목록에서 같은 방식으로 동작하게 했습니다.

## 프로젝트 개요

교육청 단위의 대규모 IT 자산을 등록, 현황 조회, 상태 관리, 이력 확인, 유상 수리 처리(유상처리) 현황, 대시보드 집계로 관리하는 Spring MVC 기반 업무 시스템입니다. (유상처리 = 보증 기간 외 비용이 청구되는 수리·처리) 교육청, 학교, 부서, 사업 차수, 자산 상태에 따라 조회 범위가 달라지는 점이 핵심이었습니다.

저는 자산 대시보드, 자산현황, 유상처리 현황, 정기 집계(배치) 이력, 검색 조건, 숫자·일자 포맷 등 운영자가 자산 상태를 확인하는 화면과 관련 SQL을 개발하고 개선했습니다.

## 맡은 역할

- JSP/Spring MVC/MyBatis 기반 자산관리 화면과 조회 SQL 개발
- 자산관리 대시보드, 자산현황, 유상처리 현황 화면 개선
- 교육청, 학교, 부서, 사업 차수, 자산 상태 기준 검색 조건 처리
- 대시보드 클릭 라우팅, 동적 컬럼, 상태별 집계, 배치 이력 화면 개선
- 운영 중 발견된 검색 조건 누락, 표시 포맷, 메뉴 구조 문제 대응

## 해결해야 했던 문제

자산관리 시스템은 권한과 기관 구조에 따라 볼 수 있는 데이터 범위가 달라집니다. 교육청 담당자는 관할 학교와 부서의 현황을 함께 봐야 하고, 학교나 부서 담당자는 자신에게 연결된 자산만 확인해야 합니다.

또한 자산 상태가 대시보드, 목록, 상세, 이력 화면에서 같은 기준으로 보여야 했습니다. 검색 조건이 누락되거나 SQL 조건이 화면 필터와 다르면 운영자가 보는 집계 수치와 실제 목록이 어긋날 수 있었습니다.

## 구현한 기능

### 자산관리 대시보드

- 사업 차수, 기관, 자산 상태별 집계 화면 개선
- 대시보드 카드 클릭 시 관련 현황 화면으로 이동하는 흐름 정리
- 기관별 대시보드 조회 조건과 표시 항목 개선
- 상태별 수량과 목록 조회 기준이 일치하도록 SQL 조건 점검

### 자산현황 및 검색 조건

- 자산현황표 데이터와 그리드 표시 컬럼 개선
- 기관, 사업 차수, 자산 상태, 제품 등 검색 조건 추가 및 누락 수정
- 화면에 보이는 필터와 MyBatis 조회 조건이 같은 기준으로 동작하도록 수정
- 숫자, 일자, 상태값 포맷을 운영자가 읽기 쉬운 형태로 정리

### 유상처리 현황과 이력

- 유상처리 상태를 별도로 확인할 수 있는 현황 화면 구성
- 처리 상태별 목록 조회와 상세 확인 흐름 개선
- 자산 메뉴 개편 과정에서 현황 메뉴를 분리하고 접근 흐름 정리
- 배치 집계 이력과 자산 변경 이력을 확인할 수 있는 화면 개선

## 기술적으로 신경 쓴 부분

권한 조건은 화면에서만 제한하면 안 되고 SQL 조회 조건까지 함께 맞아야 합니다. 교육청, 학교, 부서 권한별로 같은 검색 조건을 넣었을 때 목록, 상세, 대시보드 집계가 같은 기준으로 나오는지 확인했습니다. 권한 조건이 영향을 주는 3~5개 조회·집계 화면에서, 화면 필터에는 권한이 걸렸으나 집계 SQL에는 빠져 상위기관 수치가 부풀어 보이던 정합성 문제를 몇 건 재현·수정했습니다.

대시보드는 집계 수치를 먼저 보여주지만, 사용자는 결국 상세 목록으로 들어가 원인을 확인합니다. 그래서 3~5개 지표 카드의 클릭, 목록 이동, 필터 유지, SQL 조건이 서로 어긋나지 않도록 화면 흐름과 조회 조건을 함께 점검했습니다.

상태값과 포맷도 운영 품질에 영향을 줍니다. 자산 상태, 처리 상태, 일자, 수량이 화면마다 다르게 보이면 운영자가 같은 데이터를 다르게 해석할 수 있어 표시 기준을 정리했습니다.

## 검증 방식

- 교육청, 학교, 부서 권한별로 같은 조건을 조회했을 때 목록 범위가 올바르게 제한되는지 확인
- 대시보드 집계 수치와 클릭 이후 목록 조회 결과가 같은 기준으로 산출되는지 점검
- 자산 상태 변경 이후 목록, 상세, 이력 화면에 같은 상태가 반영되는지 확인
- 사업 차수, 기관, 자산 상태 검색 조건이 SQL 조회 조건에 빠짐없이 전달되는지 확인
- 배치 이력과 대시보드 집계 결과가 운영자가 확인할 수 있는 방식으로 표시되는지 확인

## 배운 점

B2G 업무 시스템에서는 권한과 조회 조건이 화면 품질의 핵심이라는 점을 배웠습니다. 버튼이 정상 동작해도 사용자가 볼 수 있는 데이터 범위가 잘못되면 실제 운영 수치가 틀어질 수 있기 때문입니다.

이 프로젝트를 통해 화면 필터, Controller/Service, MyBatis SQL, 집계 결과를 함께 보며 검증하는 습관을 더 강화했습니다. 특히 대시보드와 현황 화면처럼 수치가 중요한 기능에서는 "보이는 값"과 "조회 기준"을 끝까지 맞추는 일이 중요했습니다.

## 사용 기술

Java, Spring MVC, Spring Security, JSP, MyBatis, MariaDB, Tabulator, REST API, SVN
