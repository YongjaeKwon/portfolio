# 협력사 포탈 시스템

> 공개용 정리입니다. 내부 코드, 고객사별 상세 정책, 운영 URL, 계정, 환경 정보는 제외했습니다.

- 문제: 교육 대상자, 메일 발송, 제출 현황 조회 기준이 화면마다 달라질 수 있었습니다.
- 담당: Vue 화면, Tabulator 그리드, Spring Boot API, MyBatis 조회 조건, 메일·엑셀·인증 연동을 함께 구현했습니다.
- 결과: 교육 등록→대상자 업로드→메일 발송→제출 현황 조회를 하나의 운영 흐름으로 연결했습니다.

## 프로젝트 개요

수십 개 협력사가 사용하는, 협력사·고객사 담당자·교육·계약·공지·설문 데이터를 관리하는 Spring Boot 3 기반 운영 포탈입니다. 운영자가 반복적으로 처리하는 교육 등록, 대상자 관리, 엑셀 업로드, 메일 발송, 제출 현황 조회, 공지 확인, 댓글 관리 업무를 Vue 화면과 Spring Boot API로 연결했습니다.

이 프로젝트는 단순 관리자 화면보다 업무 단계가 이어지는 흐름이 중요했습니다. 예를 들어 교육관리에서는 교육을 등록한 뒤 대상자를 엑셀로 업로드하고, 대상자에게 메일을 발송한 다음 제출 현황을 다시 조회해야 했습니다. 저는 이 흐름에서 화면 상태, API 응답, MyBatis 조회 조건이 같은 기준으로 동작하도록 구현하고 점검했습니다.

대표 프로젝트로 보는 이유는 화면 구현, API 연동, SQL 조건, 파일/메일/인증 연동, 운영 이슈 대응이 한 프로젝트 안에 함께 있기 때문입니다. 프론트엔드 작업으로 시작했지만 실제로는 사용자가 보는 버튼과 그리드 상태가 서버 처리 결과, 발송 이력, 제출 현황, 권한 조건과 맞아야 했습니다.

## 맡은 역할

- Vue.js 기반 관리자 화면과 Tabulator 그리드 처리 구현
- Spring Boot Controller/Service와 MyBatis Mapper 수정
- 교육관리, 고객사 담당자 현황, 게시판/설문 댓글, 공지 읽음 이력, 협력사 검색, 인증 예외 처리 기능 개발
- 엑셀 업로드/다운로드, 메일 발송, 파일 처리처럼 운영 중 실패 케이스가 생기기 쉬운 기능의 상태 처리
- 포탈 전반의 화면을 두루 담당하며 기능 개선과 운영 이슈에 대응

## 특히 공들인 작업 3가지

### 교육 운영 흐름 연결

교육관리 기능은 화면 하나를 추가하는 작업이 아니라 운영자가 실제로 반복하는 업무 순서를 시스템 안에서 끊기지 않게 만드는 작업이었습니다. 교육 등록 이후 대상자를 업로드하고, 업로드 결과를 확인한 뒤 메일을 발송하고, 제출 현황을 다시 조회해야 했습니다.

이 흐름에서 제가 중점적으로 본 것은 세 가지였습니다.

- 업로드된 대상자와 저장된 대상자가 같은 기준으로 조회되는지
- 메일 발송 대상과 제출 현황 조회 대상이 어긋나지 않는지
- 화면의 선택 상태, API 파라미터, MyBatis 조회 조건이 같은 교육/대상자 기준을 사용하는지

단순히 버튼을 배치하는 것보다 운영자가 “지금 누구에게 발송했고, 누가 제출했는지”를 다시 확인할 수 있는 구조가 중요했습니다. 그래서 엑셀 업로드 검증, 대상자 저장, 발송 결과, 제출 현황 조회, 엑셀 다운로드까지 한 업무 흐름으로 보며 구현했습니다.

### 댓글 기능 공통화

게시판, 설문, 제안하기 3개 화면에는 비슷한 댓글/대댓글 기능이 필요했습니다. 화면마다 따로 구현하면 등록, 수정, 삭제, 대댓글 조회 방식이 조금씩 달라지고 유지보수가 어려워질 수 있었습니다.

공통 Vue 컴포넌트로 분리하면서 화면별로 달라지는 값은 입력값으로 넘기고, 댓글 조회/등록/수정/삭제 흐름은 같은 방식으로 처리하도록 정리했습니다. 이 작업은 UI 재사용뿐 아니라 운영 중 오류가 발생했을 때 한 흐름을 기준으로 점검할 수 있게 만든다는 점에서 의미가 있었습니다.

### 인증 예외와 운영 조건 처리

2단계 인증과 접근 예외 처리는 사용자의 접근 상태가 달라지는 기능이었습니다. 단순히 인증 화면을 보여주는 것보다, 로그인 사용자 유형, 접근 예외 조건, 인증 완료 여부가 서버 분기와 화면 접근 조건에서 같은 기준으로 동작해야 했습니다.

이 부분은 특정 사용자 조건에서만 문제가 드러날 수 있어 테스트 케이스를 나누어 확인했습니다. 인증이 필요한 사용자와 예외가 적용되는 사용자 조건에서 화면 접근과 서버 응답이 어긋나지 않도록 점검했습니다.

## 해결해야 했던 문제

운영자는 여러 화면을 오가며 교육 대상자, 협력사, 고객사 담당자, 제출 상태를 확인해야 했습니다. 화면마다 검색 조건과 상태값이 조금씩 달라 같은 데이터인데도 결과가 다르게 보이거나, 업로드·발송·제출 조회가 자연스럽게 이어지지 않는 문제가 생길 수 있었습니다.

또한 게시판, 제안하기, 설문처럼 댓글 기능이 필요한 화면이 여러 곳에 있어 비슷한 로직이 반복될 가능성이 컸습니다. 인증과 접근 예외처럼 로그인 이후 사용자의 접근 상태가 달라지는 기능도 화면 분기와 서버 상태가 함께 맞아야 했습니다.

특히 운영 포탈에서는 오류가 눈에 띄게 터지는 경우보다 “조회 결과가 조금 다르게 보이는 문제”가 더 위험했습니다. 발송 대상 수, 제출 완료 수, 검증 상태, 권한별 조회 범위가 화면마다 다르면 운영자는 잘못된 대상에게 메일을 보내거나 제출 현황을 잘못 판단할 수 있습니다. 그래서 기능 구현 후에는 화면에 보이는 값이 어떤 SQL 조건으로 만들어지는지까지 확인했습니다.

## 구현한 기능

### 교육관리 흐름

- 교육 등록 및 교육 대상자 조회 화면 구현
- 대상자 엑셀 업로드(회차당 수십 명 규모), 업로드 양식 다운로드, 업로드 데이터 검증 흐름 개선
- 대상자 저장, 제출 여부 조회, 교육 상세 현황 엑셀 다운로드 기능 구현
- 대상자별 안내 메일 발송 및 발송 결과 확인 흐름 추가
- 부서, 고객사 담당자, 교육센터, 제출 상태 등 운영 조건에 맞춘 조회 조건 처리
- 업로드 대상, 발송 대상, 제출 현황 조회 대상이 같은 기준으로 이어지는지 점검

### 고객사 담당자 및 협력사 현황

- 고객사 담당자 현황 조회 조건, 검증 상태 필터, 담당 업무 필수값 처리 개선
- 고객사 담당자 현황 메일 다건 발송 기능과 발송 이력 저장 흐름 구현
- 협력사 등록/수정 화면에서 검색 조건, 담당자 정보, 정산 유형, 입력 검증 개선
- 센터 검색 공통 컴포넌트와 협력사 검색 흐름 정리

### 댓글·대댓글 공통화

- 게시판, 설문, 제안하기 화면에서 반복되던 댓글/대댓글 처리 흐름을 공통 Vue 컴포넌트로 분리
- 댓글 조회, 대댓글 조회, 등록, 수정, 삭제 흐름을 한 방식으로 정리
- 파일 등록 모달과 댓글 영역의 화면 상태가 서로 충돌하지 않도록 처리

### 공지, 설문, 제안하기

- 공지사항 읽음 이력 저장과 읽음 사용자 목록 조회 기능 추가
- 설문 응답 버튼, 마감 이후 수정 가능 여부, 삭제 버튼 상태 등 운영 조건에 따른 화면 분기 개선
- 제안하기 등록/수정 검증과 메일 발송 흐름 개선

### 인증 및 운영 예외 처리

- 2단계 인증 상태와 로그인 이후 화면 접근 분기 개선
- 특정 접근 조건에서 인증 예외가 필요한 운영 케이스 처리
- 사용자 유형별 로그인 조건과 인증 상태가 어긋나지 않도록 서버 분기와 화면 접근 흐름 점검

## 기술적으로 신경 쓴 부분

화면에 보이는 조건과 SQL 조회 조건이 다른 기준으로 동작하지 않도록 확인했습니다. 관리자 화면에서는 검색 조건 하나가 누락되어도 운영자가 보는 대상자 수, 제출 상태, 발송 대상이 달라질 수 있기 때문에 Vue 상태, API 파라미터, MyBatis 조건을 함께 보며 수정했습니다.

메일 발송과 엑셀 업로드는 요청이 성공해도 실제 처리 결과가 다를 수 있는 기능입니다. 그래서 요청 버튼만 구현하지 않고, 업로드 검증 결과, 발송 대상, 발송 이력, 결과 조회까지 운영자가 확인할 수 있는 흐름을 함께 고려했습니다.

댓글 기능은 화면별로 따로 구현하면 유지보수가 어려워질 수 있어 공통 컴포넌트로 분리했습니다. 이 과정에서 작성, 수정, 삭제, 대댓글 조회 흐름을 여러 화면에서 재사용할 수 있도록 정리했습니다.

또한 운영 기능은 “요청 성공”과 “업무 완료”가 다를 수 있다는 점을 의식했습니다. 예를 들어 메일 발송 API가 호출되었다고 해서 운영자가 필요한 확인이 끝난 것은 아닙니다. 어떤 대상에게 발송했는지, 실패한 대상이 있는지, 이후 제출 현황이 어떤 기준으로 조회되는지까지 확인할 수 있어야 했습니다.

이 때문에 화면 상태는 가능하면 업무 상태와 맞춰 나눴습니다. 선택 전, 업로드 후 검증, 저장 완료, 발송 요청, 발송 결과 확인, 제출 현황 조회처럼 운영자가 이해할 수 있는 단위로 상태를 구분하는 데 신경 썼습니다.

## 검증 방식

- 교육 등록, 대상자 업로드, 메일 발송, 제출 현황 조회가 같은 대상자 기준으로 이어지는지 확인
- Tabulator 행 선택, 필터, 페이징, 엑셀 다운로드 조건이 API 조회 조건과 일치하는지 점검
- 고객사 담당자 검증 상태, 부서, 센터, 협력사 조건을 바꿨을 때 화면 결과와 SQL 결과가 같은지 확인
- 인증 상태, 접근 예외 조건, 로그인 사용자 유형에 따라 접근 가능한 화면이 달라지는지 확인
- 댓글/대댓글 공통 컴포넌트가 게시판, 설문, 제안하기 화면에서 같은 방식으로 동작하는지 확인
- 업로드/발송/조회 기능에서 성공, 실패, 빈 결과, 권한 조건 변경 케이스를 나누어 확인
- 엑셀 다운로드 결과가 화면 필터와 같은 기준으로 생성되는지 비교

## 배운 점

운영 포탈의 화면 개발은 버튼과 그리드를 배치하는 일보다 업무 기준을 정확히 맞추는 일이 중요했습니다. 특히 교육 대상자, 메일 발송, 제출 현황처럼 같은 데이터가 여러 단계를 거치는 기능에서는 화면 상태와 서버 조회 조건이 조금만 어긋나도 운영자가 잘못된 판단을 할 수 있습니다.

이 프로젝트를 통해 Vue 화면, Spring Boot API, MyBatis SQL, 메일/엑셀/인증 연동을 함께 보며 개발하는 경험을 쌓았습니다. 프론트엔드 개발자라도 데이터 흐름과 업무 기준을 이해해야 사용자가 실제로 믿고 쓸 수 있는 화면을 만들 수 있다는 점을 배웠습니다.

이 프로젝트는 운영자가 반복 처리하는 교육/발송/제출 확인 업무를 화면, API, SQL, 발송 이력까지 연결한 운영 포탈 경험으로 정리할 수 있습니다.

## 사용 기술

Vue.js, JavaScript, Tabulator, Chart.js, Java, Spring Boot, Spring Security, MyBatis, MariaDB, Git
