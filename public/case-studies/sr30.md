# SR30 물류관리시스템

> 공개용 정리입니다. 내부 코드, 운영 URL, 계정, 환경 정보는 제외했습니다.

## 프로젝트 개요

Nexacro와 Spring MVC 기반의 물류·서비스 운영 시스템입니다. 약 1년간 일정, 설문, 물류, 재고, 서비스 리포트, KPI, 엑셀 다운로드, 관리자 이력 조회처럼 운영자가 반복적으로 확인하는 20여 개 화면을 개발·개선하고, 100여 건 규모의 운영 이슈와 기능 개선에 대응했습니다.

이 프로젝트는 최신 프론트엔드 기술보다 운영 시스템의 업무 조건을 정확히 이해하는 경험이 컸습니다. 화면 필터, SQL 조건, 통계 집계, 엑셀 다운로드 결과가 같은 기준으로 동작하도록 점검하는 일이 많았습니다.

## 맡은 역할

- Nexacro 기반 운영 화면 신규 개발 및 기능 개선
- Spring MVC/MyBatis 기반 조회 조건, 통계 SQL, 엑셀 다운로드 조건 수정
- 일정 공유, 근무 일정, 설문, 물류/재고, 서비스 리포트, KPI 화면 개선
- 개인정보 다운로드 이력, 데이터 삭제 이력, 접근 이력 등 관리자 이력 조회 기능 대응
- 운영 중 발견된 검색 조건 누락, 화면 표시 오류, 데이터 중복 조회 문제 수정

## 해결해야 했던 문제

운영자가 보는 통계와 리포트는 업무 판단의 기준이 됩니다. 따라서 화면 필터와 SQL 조건이 다르면 같은 기간이나 센터를 조회해도 화면마다 다른 결과가 나올 수 있습니다.

또한 물류, 재고, 일정, 서비스 리포트처럼 업무 영역이 넓어 화면마다 조건과 상태값이 조금씩 달랐습니다. 새로운 기능을 만들 때도 기존 화면과 같은 기준으로 조회되고 엑셀 결과까지 일치하는지 확인해야 했습니다.

## 구현한 기능

### 일정 및 근무 관리

- 일정 공유 화면과 근무 일정 관리 화면 개선
- 센터, 담당자, 기간 조건에 따른 조회 조건 처리
- 운영자가 일정 상태를 확인하고 후속 업무를 진행할 수 있도록 화면 흐름 정리

### 설문 및 통계

- 설문 등록, 대상자 관리, 응답 저장, 통계 조회 화면 개선
- 브랜드별 운영 설문 화면 대응
- 응답 상태와 통계 집계 결과가 같은 기준으로 계산되도록 SQL 조건 점검
- 설문 관련 엑셀 다운로드 조건과 화면 필터 일치 여부 확인

### 물류, 재고, 자재, 리포트

- 물류/재고/자재 조회 화면의 검색 조건과 표시 항목 개선
- 서비스 리포트, SLA 리포트, KPI 등 운영 리포트 화면 개선
- 설치/철거, 사후 업무 대시보드와 현황 화면 조회 조건 대응
- 센터/거래처 검색 조건과 결과 표시 흐름 개선

### 관리자 이력

- 개인정보 다운로드 이력 조회
- 데이터 삭제 이력 조회
- 접근 이력과 관리자 로그 관련 화면 개선
- 이력 화면의 검색 조건, 기간 조건, 엑셀 다운로드 결과 점검

## 기술적으로 신경 쓴 부분

레거시 운영 시스템에서는 화면 하나만 고치면 끝나는 경우가 드물었습니다. 목록 화면, 상세 화면, 통계 화면, 엑셀 다운로드가 같은 SQL 조건을 공유하거나 비슷한 조건을 따로 가지고 있어, 한 건을 고칠 때 연관된 2~3개 유사 화면까지 함께 점검해야 했습니다.

특히 통계와 KPI 화면은 수치의 신뢰성이 중요했습니다. 화면에서 선택한 센터, 기간, 상태 조건이 SQL 집계 기준과 엑셀 다운로드 결과에 동일하게 적용되는지 확인했습니다.

이력 조회 화면은 운영 감사와 연결될 수 있어 검색 조건과 결과 누락을 더 신중히 봤습니다. 개인정보 다운로드, 데이터 삭제, 접근 이력처럼 민감한 정보는 누가, 언제, 어떤 조건으로 확인했는지 추적할 수 있어야 했습니다.

## 검증 방식

- 같은 센터, 기간, 상태 조건으로 화면 조회와 엑셀 다운로드 결과가 일치하는지 확인
- 설문 응답 상태와 통계 집계 결과가 같은 기준으로 계산되는지 점검
- 리포트/KPI 화면에서 검색 조건 누락이나 중복 조회가 없는지 확인
- 일정, 재고, 서비스 리포트 등 유사 화면에서 같은 조건이 같은 방식으로 동작하는지 확인
- 관리자 이력 화면에서 기간, 사용자, 업무 조건에 따른 조회 범위가 올바른지 점검

## 배운 점

SR30을 통해 운영 시스템에서 가장 중요한 것은 "사용자가 보는 기준이 일관되는가"라는 점을 배웠습니다. 화면은 달라도 업무 기준이 같다면 조회 조건, 통계 기준, 다운로드 결과도 같아야 합니다.

또한 레거시 환경에서도 프론트엔드 개발자는 화면만 보지 않고 데이터 조건, SQL, 엑셀 결과, 이력 조회까지 함께 확인해야 한다는 점을 체감했습니다. 이 경험은 이후 Vue 기반 관리자 화면과 B2B/B2G 업무 시스템을 개발할 때도 큰 기반이 되었습니다.

## 사용 기술

Nexacro, Java, Spring MVC, MyBatis, PL/SQL, Oracle, Excel, SVN
